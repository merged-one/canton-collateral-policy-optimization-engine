# Risk Register

| ID | Risk | Why It Matters | Initial Mitigation | Evidence Needed | Status |
| --- | --- | --- | --- | --- | --- |
| R-001 | Eligibility or haircut policy is underspecified or non-deterministic. | Different parties could derive different coverage decisions from the same inputs. | Define versioned policy inputs and deterministic evaluation rules before implementation, starting with a strict CPL schema and explicit selection semantics. | CPL spec, schema, ADRs, invariant mapping, test vectors | Open |
| R-002 | Confidential information leaks across parties or reports. | Margin and collateral data may be commercially sensitive. | Keep confidentiality in the architecture boundary, use explicit Daml stakeholder sets for workflow contracts, define role-specific report profiles, and track the resulting exposure surface in the threat model. | privacy model, ADR 0007, Daml mapping, access-control tests, report review | Open |
| R-003 | Substitution or return workflows are non-atomic. | Exposure could become temporarily uncovered or duplicated. | Treat atomic workflow semantics as a top-level invariant, separate workflow requests from settlement instructions, and prove the skeleton path with Daml scripts before adding integrations. | workflow specs, ADR 0007, Daml test plan, execution evidence | Open |
| R-004 | Double-encumbrance or replay bugs appear under retries or concurrency. | The same collateral could back multiple obligations incorrectly. | Model encumbrance and replay handling explicitly in the workflow contracts and extend the current script checks toward replay and concurrency scenarios. | invariants, Daml mapping, Daml tests, replay tests | Open |
| R-005 | Execution reports diverge from committed workflow state. | Operators and reviewers could trust incorrect evidence. | Derive reports from workflow choices, keep report templates separate from operator roles, and verify report emission through Daml scripts and prompt evidence. | report schema, Daml report module, demo artifacts, audit checks | Open |
| R-006 | Dependency setup becomes non-reproducible. | Demos and verification would be hard to trust or repeat. | Require pinned versions, checksum-verified bootstrap downloads, and explicit bootstrap commands through ADRs. | setup docs, bootstrap script, reproducible commands, lockfiles later | Open |
| R-007 | Repository changes outpace documentation and evidence. | The safety case would erode even if code seems to work. | Enforce worklog, ADR, invariant, and evidence updates on every feature. | worklog, ADRs, evidence manifest | Open |
| R-008 | Architecture assumptions do not match Canton Quickstart or token-standard realities. | Rework could invalidate early design and reports. | Pin versions, record interface assumptions, and validate integration surfaces before logic. | dependency ADRs, integration docs, prototypes later | Open |
| R-009 | Proposal scope broadens faster than implementation sequencing can support. | The repo could look comprehensive while lacking depth in the highest-risk layers. | Keep milestone gating explicit and distinguish roadmap intent from completed implementation. | roadmap, ADRs, worklog, release checklist | Open |
| R-010 | CPL omits wrong-way-risk, cross-currency, or jurisdictional controls that real workflows require. | Early schemas could force breaking redesign once real policies are modeled. | Treat policy-profile coverage and control-field completeness as Milestone 1 design gates and publish market-practice example profiles before engine work starts. | CPL spec, schema, example policies, ADRs, test strategy | Open |
| R-011 | Policy, optimization, workflow, and reporting concerns collapse during implementation. | Determinism, explainability, and atomicity would become difficult to test or defend. | Lock the six-boundary architecture into ADRs, architecture docs, invariant mappings, and the first Daml contract split before deeper implementation begins. | architecture docs, ADRs, Daml mapping, interface contracts | Open |
| R-012 | LocalNet overlays or demo tooling acquire hidden business authority. | Prototype results could depend on environment shortcuts instead of the documented control plane. | Keep runtime responsibilities explicit and require evidence that reports come from committed state. | deployment model, runbooks, prompt evidence, scenario evidence later | Open |
| R-013 | Report profiles over-disclose confidential workflow or inventory detail. | Audit or integration outputs could leak commercially sensitive information. | Treat the current workflow-party execution reports as a baseline only, define role-specific disclosure profiles next, and test them against the Daml stakeholder model before widening access. | privacy model, ADR 0007, report schema, security review | Open |
| R-014 | CPL consumers drift in version handling or silently accept unsupported extensions. | Different loaders could accept materially different policy shapes while appearing to process the same package. | Separate `cplVersion` from `policyVersion`, reject undeclared fields, and validate published policies before load. | ADR 0005, CPL schema, validation command, conformance tests later | Open |
| R-015 | Policy evaluation reports rely on normalized valuation inputs before reference-data and FX contracts are pinned. | Absolute concentration limits or lendable values could be misread if consumers assume the engine performs hidden conversion. | Keep the engine input explicit about `valuationCurrency`, reject unsupported absolute-threshold currencies, document the normalized-input assumption in the report spec and ADR, and pin reference-data contracts next. | policy evaluation report spec, ADR 0008, policy-engine tests, future reference-data contracts | Open |
| R-016 | Repository naming or acronym drift could blur subsystem boundaries or create market-structure ambiguity. | Contributors could incorrectly treat the repository as only a policy-plus-optimization service or confuse it with a CCP-like market role. | Use "Canton Collateral Control Plane" on primary identity surfaces, document the control-plane versus data-plane split explicitly, retain the former name only as a deprecated historical alias, and forbid new acronyms such as `CCCP` or `CCP` for the repository. | ADR 0010, README, glossary, architecture docs | Open |
| R-017 | Optimizer output could create unnecessary churn or hidden objective drift. | Operators could substitute collateral without real economic benefit or be unable to defend why one feasible set was chosen over another. | Publish the optimizer objective explicitly, keep the search deterministic, preserve a no-churn rule for economically equal current sets, and emit explanation traces plus machine-readable substitution deltas. | ADR 0009, optimization report spec, economic note, optimizer tests | Open |
